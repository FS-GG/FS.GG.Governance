# Quickstart: Validating `fsgg refresh`

This guide proves `fsgg refresh` end to end against a real temporary repository. It validates the three
user stories and the key guarantees. Details live in [contracts/cli.md](./contracts/cli.md),
[contracts/manifest.md](./contracts/manifest.md), [contracts/refresh.schema.md](./contracts/refresh.schema.md),
and [data-model.md](./data-model.md).

## Prerequisites

- .NET `net10.0` SDK; repo builds clean (`dotnet build FS.GG.Governance.sln`, `TreatWarningsAsErrors=true`).
- The two new projects present: `src/FS.GG.Governance.RefreshCommand`, `src/FS.GG.Governance.RefreshJson`.

## Build & test

```bash
dotnet build FS.GG.Governance.sln
dotnet test tests/FS.GG.Governance.RefreshJson.Tests
dotnet test tests/FS.GG.Governance.RefreshCommand.Tests
```

Expected: solution builds clean; both suites green. Surface baselines match
(`surface/FS.GG.Governance.RefreshCommand.surface.txt`, `…RefreshJson.surface.txt`); regenerate
intentionally with `BLESS_SURFACE=1 dotnet test <project>`.

## Scenario 1 — bring a stale view current (US1, SC-001/SC-002)

1. In a temp repo with a `.fsgg/refresh.yml` declaring one view whose sources are all current, run
   `fsgg refresh`. **Expect**: nothing regenerated, every view reported current, **exit 0**
   (`nothing-to-refresh`).
2. Change one declared source so its view goes stale; re-run `fsgg refresh`. **Expect**: exactly that view
   regenerated (others byte-for-byte untouched), reported `regenerated` with the drifted category, the
   regenerated bytes equal what the declared generator produces, **exit 5** (`regenerated`).
3. Re-run once more with no further change. **Expect**: nothing regenerated, **exit 0** — staleness is by
   digest, not file presence (recorded provenance now matches).
4. Run against a repo with an **empty** `refresh.yml`. **Expect**: "nothing to refresh," **exit 0**
   (empty manifest is not an error — FR-012).

## Scenario 2 — preview without writing (US2, SC-003)

1. With a stale view present, run `fsgg refresh --dry-run`. **Expect**: the view reported
   `would-regenerate` with its drifted source(s) and reason; the working tree **byte-for-byte unchanged**
   afterward; **exit 5**.
2. Run `fsgg refresh --dry-run` over an all-current repo. **Expect**: all-current, nothing written,
   **exit 0**.
3. Assert (no-mutation guard) that a `--dry-run` over a stale repo writes **no** view, lock, or artifact.

## Scenario 3 — deterministic `refresh.json` (US3, SC-004)

1. Run `fsgg refresh --refresh-out out.json` (or `--dry-run --refresh-out`) twice over identical state with
   identical outcomes. **Expect**: the two `out.json` files are **byte-for-byte identical**.
2. Run with `--json`. **Expect**: stdout equals the persisted `refresh.json` verbatim (one source of truth).
3. Inspect `refresh.json`. **Expect**: `schemaVersion = "fsgg.refresh/v1"`; no timestamp, absolute path,
   username, or machine-specific content; a `stale-unresolved` view (if any) carries a `reason` and is never
   reported `current`.

## Failure & guarantee checks (SC-005/SC-006/SC-007)

- **InputUnavailable (3)**: absent/malformed `refresh.yml` or an absent declared source ⇒ exit 3 with a
  diagnostic naming the offending file; no partial regeneration.
- **ToolError (4)**: a declared generator that exits non-zero (or an unwritable output path) ⇒ exit 4, no
  partial view left behind, distinct from `stale-unresolved` (1).
- **StaleUnresolved (1)**: a stale view whose source cannot be resolved ⇒ exit 1, reported
  `stale-unresolved`, never fabricated `current`.
- **UsageError (2)**: `--view-kind X --view Y` together, or an unknown flag ⇒ exit 2, nothing written.
- **Product-neutral (SC-006)**: grep the two projects — no product/view/path/generator/renderer identity;
  all come from `refresh.yml`.
- **Network-free (SC-007)**: the scope-guard test asserts the command's own assemblies reference no network
  API.
- **Not the merge authority (FR-017)**: a clean `fsgg refresh` performs no merge gating.
