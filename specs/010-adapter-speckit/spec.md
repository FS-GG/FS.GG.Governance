# Feature Specification: The Spec Kit Adapter — Governance Dogfoods This Repo's Own Workflow As Data

**Feature Branch**: `010-adapter-speckit`

**Created**: 2026-06-18

**Status**: Draft

**Input**: User description: "create specs for the next item in the project." (resolved to F10 · `010-adapter-speckit` from the dated Spec Kit implementation plan)

**Change Classification**: **Tier 1** — introduces the **first concrete production adapter**: the **Spec Kit adapter**, the domain whose vocabulary is this repository's own Spec Kit workflow (`constitution → specify → clarify → plan → tasks → analyze → implement → merge`). It supplies — through the already-shipped F09 SPI — *only* its own facts (phases, artifacts, task states, the dependency topology), an artifact mapping, probes, a phase-check rule catalog, and the single merge fence; everything cross-cutting is reused from the kernel (F01–F07). It adds a new curated public signature contract and a new surface-area baseline. It is a **pure** value/fold layer — the sensing of the live repository (reading `.specify/feature.json`, parsing `tasks.md`, etc.) is the F08 effects shell / F12 CLI's job, not this feature — so **Constitution Principle IV is N/A** (no state, no I/O). It is the first of the two unrelated domains that meet **Milestone M3 — the adoption bar**: governance dogfoods **this repo's own** Spec Kit lifecycle.

## User Scenarios & Testing *(mandatory)*

The "users" of this feature are the **maintainers of this repository** running their own Spec Kit lifecycle, and, one level down, the **kernel itself** as the observer attached to that lifecycle. Spec Kit is the **harness** — the authored, agent-run work loop. Governance is the **observer**: it never generates `spec.md`, `plan.md`, or `tasks.md`; it reads them as artifacts, asserts facts, and runs rules. That keeps the anti-goal ("do not replace authored Spec Kit artifacts") **structural, not aspirational** — the adapter has no authoring surface at all, only a sensing/checking one.

This feature is the **first concrete adopter** of the F09 adapter SPI. Where F09 shipped the generic machinery and proved it with two neutral toy domains, F10 makes the adoption bar real: the Spec Kit workflow plugs into the kernel by supplying **exactly** its own five components — a closed `SpecKitFact` union (phases, artifact presence, task states, task dependencies, skill bindings, constitution-area completeness), a `SpecKitArtifact → ArtifactRef` mapping, the probes its rules compose, the **phase-check rule catalog**, and the **single merge fence** — and gets inference, three-valued verdicts, the reified `Check` algebra and its interpreters, `CheckTier` arbitration, the evidence/taint DAG, JSON explanation and contract, severity, and routing/run-modes **for free**. The adapter re-implements **none** of those.

The keystone behaviours are four, each a direct translation of the Spec Kit design (`docs/governance-design/speckit-in-the-system.md`): **(1)** the stateless kernel governs a **stateful lifecycle** by treating the current phase as a *supplied fact*, so phase-guarded rules contribute only once the feature has reached their phase; **(2)** the previous design's monolithic `analyze` pass and its pile of always-blocking gates become a **catalog of reified rules**, each declaring *who decides* (`CheckTier`) and *whether it blocks* (`Severity`), each rendering to a sentence and explaining itself; **(3)** **nothing blocks before merge** — the inner-loop phases run advisory; **merge is the single fence** that flips to `Gate`, recomputes from the base branch, and lets blocking rules bite; and **(4)** the **constitution is the dial** — the constitution phase is where the fences and severities are authored, a small reviewable place to opt back into hard guarantees, and the Constitution Check just verifies the dial was filled in honestly. The `tasks.deps.yml` topology becomes `TaskDependsOn` facts and the evidence/synthetic-taint model runs the **kernel** over them — `EvidenceGraph` is a derivation, **not a bespoke engine**.

This feature builds directly on **F09** (it is authored against the adapter `Adapter<'fact,'artifact,'change>` SPI record and the `Lift`/`Composition` machinery — it supplies the five components and the F04 `Bridge` kernel wiring, and composes at a root alongside F11 when a project uses both) and through F09 on **F07** (its rules carry `CheckTier`/`Severity`, and its `mergeFence` is an F07 `Fence` that flips the run-mode to `Gate`), **F05** (its `TaskState`/`TaskDependsOn` facts are the evidence/taint model run unchanged), **F04** (each rule is a `CheckRule<SpecKitFact>` — a `Check` + `CheckTier` + `Severity`), **F03** (its probes are reified `Check`s over `Probe`/`ArtifactRef`/`Outcome`), and **F01–F02** (the `Rule`/`FactSet`/`FixedPoint` and the verdict algebra). It is **pure** — values and total folds, no `Model`/`Msg`/`Effect`, no interpreter — so Principle IV does not apply; the impure effects shell (**F08**, already shipped) and the **F12** CLI are what **sense** the live repository into `SpecKitFact`s and wire the adapter into a running loop.

### User Story 1 - The Spec Kit workflow is governed as data: phases, artifacts, and task states are facts (Priority: P1)

A maintainer adopts the kernel for this repository's Spec Kit workflow by supplying **only** the five SPI components for the Spec Kit domain: (1) the closed `SpecKitFact` union (`PhaseReached`, `ArtifactPresent`, `TaskState`, `TaskDependsOn`, `SkillBound`, `ConstitutionArea`), (2) the `SpecKitArtifact → ArtifactRef` mapping (`Constitution`, `Spec`, `Plan`, `Research`, `DataModel`, `Contracts`, `Quickstart`, `Tasks`, `TaskDeps` onto the kernel's structural artifact reference), (3) the probes its rules compose, (4) the phase-check rule catalog, and (5) the merge fence. Inference, arbitration, evidence, rendering, hashing, explanation, severity, and run-modes are **all reused from the kernel** — the adapter writes none of them. Governance **reads** the Spec Kit artifacts as data and asserts facts; it never authors or rewrites them.

**Why this priority**: This is the thesis of the milestone made concrete for domain #1. If governing this repo's own workflow required re-implementing inference or arbitration, or an authoring surface that could overwrite `spec.md`, the boundary would be in the wrong place and the anti-goal would be aspirational rather than structural. The five-part Spec Kit adapter is the minimum viable adopter and the foundation every other story in this feature guards, lifts, or routes, so it ships first.

**Independent Test**: Author the Spec Kit adapter as a single `Adapter<SpecKitFact, SpecKitArtifact, _>` value supplying only the five components; feed it a synthetic `FactSet` of `SpecKitFact`s and confirm it derives facts, evaluates its rules, and renders an explanation using **only** kernel facilities — and that the adapter module contains **no** inference, arbitration, evidence, rendering, hashing, explanation, severity, or routing code of its own, and **no** function that writes a Spec Kit artifact.

**Acceptance Scenarios**:

1. **Given** the F09 adapter SPI, **When** the Spec Kit adapter supplies the five components (the `SpecKitFact` union, the artifact mapping, probes, the phase-check catalog, the merge fence) plus the F04 `Bridge` kernel wiring, **Then** this repo's Spec Kit workflow is fully governable with no further domain code.
2. **Given** the Spec Kit adapter, **When** its rules are evaluated over a supplied `FactSet`, **Then** every cross-cutting facility (fixed point, tier arbitration, verdict algebra, evidence/taint, render/hash/explain, routing) comes from the kernel, not the adapter.
3. **Given** the adapter, **When** it is inspected, **Then** it exposes **no** operation that generates or mutates `spec.md`/`plan.md`/`tasks.md` — governance observes artifacts as data, it does not author them (the anti-goal is structural).

### User Story 2 - Phase-guarded rules contribute only at or after their phase (Priority: P1)

The Spec Kit lifecycle is **stateful** — a feature moves `constitution → specify → clarify → plan → tasks → analyze → implement → merge` — but the kernel is **stateless**. The adapter bridges the two by treating the **current phase as a supplied fact** (`PhaseReached`), and guarding rules with `whenPhase`: `whenPhase Plan check` contributes its check only once the feature has reached `Plan`, and is silent (does not fire, does not fail) before then. A rule that asks "does `plan.md` address every requirement in `spec.md`?" is meaningless before `plan.md` exists, so the phase guard makes it a definite **not-applicable** until `Plan`, then an advisory check at/after `Plan` — never a spurious failure for an artifact that does not yet exist.

**Why this priority**: A stateless rule engine governing a staged lifecycle is the central design move of the Spec Kit adapter; without the phase guard, every rule would fire in every phase and the inner loop would be a wall of false negatives for not-yet-authored artifacts. `whenPhase` is the mechanism the rest of the catalog depends on, so it is co-equal P1 with the adapter itself.

**Independent Test**: Take a `whenPhase Plan` rule; evaluate it over a `FactSet` with `PhaseReached Specify` and confirm it contributes nothing (a definite not-applicable, not a failure); evaluate the same rule over a `FactSet` with `PhaseReached Plan` (or later) carrying the relevant artifact facts and confirm it now contributes its check verdict.

**Acceptance Scenarios**:

1. **Given** a rule guarded by `whenPhase Plan`, **When** the supplied phase is before `Plan`, **Then** the rule is inert — it contributes a definite not-applicable, not a failing or unknown verdict.
2. **Given** the same rule, **When** the supplied phase is `Plan` or later, **Then** the rule contributes its check exactly as if unguarded.
3. **Given** the current phase supplied as an ordinary `PhaseReached` fact, **When** the catalog is evaluated, **Then** the stateful lifecycle is handled entirely through the supplied fact — the adapter holds no mutable phase state of its own.

### User Story 3 - Nothing blocks before merge; merge is the single fence that flips to Gate (Priority: P1)

Every inner-loop phase runs **advisory**: `constitution`, `specify`/`clarify`, `plan`, `tasks`, `analyze`, and `implement` *inform* — the full rule catalog can run as a **report** across all tiers, and even a deterministic-and-failing graph check does not stop the work. **Merge is the single fence**: it flips run-mode to `Gate`, recomputes the verdict from the base branch, and lets the blocking rules bite — constitution complete, evidence not synthetic, contracts current, fenced surfaces verified. So everything before merge informs; only the boundary enforces. The `Sandbox` mode during `specify`/`clarify` lets a maintainer throw a draft spec around with zero friction, and they still cannot land it unchecked, because merge recomputes independently. This is the **one deliberate behaviour change** from the previous design: nothing blocks before merge, and hard-stopping a specific earlier phase is an opt-in one-line `|> blocking` plus a fence on that phase, visible in the constitution.

**Why this priority**: The old pain was plan-time and even documentation edits triggering heavy always-blocking machinery; the fix — advisory inner loop, one enforcing boundary — is the feature's headline value and the thing that makes day-to-day adoption friction-free without sacrificing the guarantee that nothing unchecked reaches the base branch. It is the run-mode mapping the whole catalog is read through, so it is P1.

**Independent Test**: Evaluate the catalog over a `FactSet` for each inner-loop phase in `Inner` mode and confirm every verdict is advisory — a failing deterministic check reports but does not produce a blocking route; then evaluate the same facts at `PhaseReached Merge` through the `mergeFence` in `Gate` mode and confirm the blocking rules now flip the route to a blocking gate.

**Acceptance Scenarios**:

1. **Given** any inner-loop phase in `Inner` mode, **When** the rule catalog is evaluated, **Then** all results are advisory — including deterministic failing checks — and the route never blocks.
2. **Given** `PhaseReached Merge`, **When** the `mergeFence` evaluates in `Gate` mode, **Then** the blocking rules (constitution complete, evidence not synthetic, contracts current, fenced surfaces verified) flip the route to a blocking gate and a failing blocking rule refuses the merge.
3. **Given** a maintainer who wants an earlier hard-stop (e.g. refuse to leave `tasks` with a cyclic graph), **When** they add a `|> blocking` severity plus a fence on that phase, **Then** that phase enforces — the default-advisory behaviour is opt-out per rule, visible in the constitution, with no change to the kernel.

### User Story 4 - The constitution is the dial that configures fences and severities (Priority: P2)

In the previous design the constitution was prose plus a separate guidance check. Here the **constitution phase is where the fences and severities are authored** — the constitution configures which surfaces are fenced and which rules block at merge. The "complete enforcement ↔ light" dial lives in `constitution.md` as a small, reviewable, single place to opt back into hard guarantees for the few things that warrant them (a Tier-1 surface change, synthetic-evidence at merge). The Constitution Check is then just a deterministic rule that verifies the dial was **filled in honestly** — the `ConstitutionArea` facts report which areas are present and non-placeholder — advisory in the inner loop, blocking at merge.

**Why this priority**: The dial is what makes the advisory-by-default posture safe: it is the explicit, reviewed surface where a maintainer opts specific guarantees back to blocking, so the lightness of the inner loop is a deliberate, visible choice rather than a silent weakening. It refines the run-mode story (P1) into a configuration model, so it ranks P2 but ships with the catalog because the merge fence's blocking set is exactly what the dial configures.

**Independent Test**: Supply `ConstitutionArea` facts marking some areas filled and one a placeholder; evaluate the Constitution Check and confirm it is advisory in the inner loop and reports the placeholder area; evaluate it at `Merge` and confirm the placeholder makes it a blocking failure — and confirm the merge fence's blocking set is the set the constitution authored, not a hard-coded list in the adapter.

**Acceptance Scenarios**:

1. **Given** the constitution authoring the fences and severities, **When** the merge fence is assembled, **Then** which surfaces are fenced and which rules block at merge is determined by the authored dial, not by fixed logic in the kernel.
2. **Given** `ConstitutionArea` facts, **When** the Constitution Check runs in the inner loop, **Then** it reports completeness advisorily; **When** it runs at `Merge`, **Then** an unfilled or placeholder area is a blocking failure.
3. **Given** the dial set to "light", **When** the inner loop runs, **Then** only the few opted-in surfaces block at merge and everything else stays advisory — the lightness is the authored configuration, visibly reviewable in one place.

### User Story 5 - The evidence and dependency graph runs through the kernel's F05 model, not a bespoke engine (Priority: P2)

The `tasks.md` task states are the kernel's **evidence facts** — `Real`, `Synthetic`, `Failed`, `Skipped` — carried as `TaskState`, and the `tasks.deps.yml` topology becomes `TaskDependsOn` facts. The computed `AutoSynthetic` taint **flows down the dependency graph** by the kernel's own forward-chaining fixed point: `EvidenceGraph` is a **derivation over the supplied facts**, not a separate validation engine the adapter ships. The well-formedness checks (every task has deps, deps resolve, the graph is acyclic, ids/skills resolve) are a deterministic rule (`allOf [...]`) — advisory while authoring, promoted to blocking only behind the merge fence. The merge fence's `evidenceNotSynthetic` rule is the blocking verdict that keeps synthetic-only work from reaching the base branch; disclosure is mandatory and **no flag changes a verdict** — honesty about evidence is enforced separately from the freedom to iterate that the run-mode hatch provides.

**Why this priority**: Reusing the F05 evidence/taint model instead of a bespoke `analyze` engine is the concrete proof that this adapter "reuses 100% of kernel facilities" — the most load-bearing check in the whole workflow (synthetic evidence must not reach `main`) is just a rule over supplied facts, not new machinery. It ranks P2 because the SPI adoption, phase guards, and run-mode mapping (P1) are the spine; this story is the evidence that even the heaviest legacy gate collapses to a kernel derivation.

**Independent Test**: Supply `TaskState` and `TaskDependsOn` facts forming a dependency chain with one `Synthetic` upstream task; evaluate the catalog and confirm the `AutoSynthetic` taint propagates down the chain via the kernel fixed point (not adapter code); confirm the graph well-formedness rule is advisory in the inner loop; and confirm `evidenceNotSynthetic` is a blocking failure at `Merge` when tainted evidence would reach the base.

**Acceptance Scenarios**:

1. **Given** `TaskState`/`TaskDependsOn` facts, **When** the catalog is evaluated, **Then** the `AutoSynthetic` taint is computed by the kernel's forward-chaining fixed point over those facts — the adapter ships no bespoke graph engine.
2. **Given** a well-formedness check (`everyTaskHasDeps`, `depsResolve`, `acyclic`, `skillIdsResolve`), **When** it runs in the inner loop, **Then** it is advisory; **When** it runs behind the merge fence, **Then** it can block.
3. **Given** synthetic-tainted evidence and `PhaseReached Merge`, **When** `evidenceNotSynthetic` evaluates in `Gate` mode, **Then** it is a blocking failure and no flag or override flips that verdict — disclosure does not buy a pass.

### Edge Cases

- **Phase before an artifact exists**: a `whenPhase Plan` rule evaluated before `Plan` is reached is a definite **not-applicable** (the antecedent phase fact is absent), never a failing or unknown verdict for an artifact that does not yet exist.
- **Phase fact absent entirely**: if no `PhaseReached` fact is supplied, every phase-guarded rule is inert (not-applicable) — the catalog does not assume a phase, and a missing phase is malformed input, not a silent default to `Merge`.
- **Inner-loop deterministic failure**: a deterministic graph check that *fails* in an inner-loop phase still only **reports** — it never blocks, because run-mode `Inner` keeps every result advisory regardless of severity.
- **Merge recompute from base**: the merge fence recomputes the verdict from the **base branch**, so a passing inner-loop run does not exempt the change — the boundary enforces independently of what the inner loop reported.
- **Cyclic task graph**: a cycle in `TaskDependsOn` is reported by the acyclicity check (advisory inner loop); it becomes a hard-stop only if the constitution opts that phase into a `|> blocking` fence.
- **Unresolved task dependency or skill id**: a `TaskDependsOn`/`SkillBound` referencing an absent task/skill is reported by the resolve checks as a definite failure of that deterministic rule, distinguishable from a not-yet-known input.
- **Constitution area left as placeholder**: an unfilled or placeholder `ConstitutionArea` is advisory while authoring and a **blocking** Constitution-Check failure at merge — the dial must be filled honestly to land.
- **Synthetic evidence reaching merge**: `Synthetic`/`AutoSynthetic`-tainted task states make `evidenceNotSynthetic` a blocking failure at `Merge`; no disclosure flag or override changes that verdict.
- **AgentReviewed advisory check**: a judgement rule (`plan-covers-spec`, "are the tasks complete and ordered?") is `Opaque`/`AgentReviewed` and **advisory** — it reports gaps during the relevant phase via its `Question`, it does not block the inner loop.
- **HumanOnly check**: an "is the feature in scope / worth doing?" check is `HumanOnly` — it routes to a person and never resolves to a deterministic verdict, exactly as the tier demands.
- **Adapter used alone vs. composed**: the Spec Kit adapter governs this repo standalone (its rules over `SpecKitFact` directly) and also lifts unchanged into a project coproduct when composed with a second domain (F11) at one root — the standalone and lifted verdicts are identical (the F09 faithful-lift guarantee).

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: The feature MUST define the **Spec Kit domain vocabulary** as a closed `SpecKitFact` union — at minimum `PhaseReached` (the supplied current phase), `ArtifactPresent`, `TaskState` (carrying an evidence state), `TaskDependsOn` (the dependency topology), `SkillBound`, and `ConstitutionArea` — together with the `SpecKitArtifact` and `Phase` enumerations the workflow ranges over.
- **FR-002**: The feature MUST provide the **artifact mapping** `SpecKitArtifact → ArtifactRef` — every Spec Kit artifact kind (`Constitution`, `Spec`, `Plan`, `Research`, `DataModel`, `Contracts`, `Quickstart`, `Tasks`, `TaskDeps`) onto the kernel's structural artifact reference — the single point where Spec Kit artifacts meet the otherwise domain-neutral algebra.
- **FR-003**: The feature MUST be authored as a single F09 `Adapter<SpecKitFact, SpecKitArtifact, _>` value supplying **exactly the five SPI components** (fact-union identity, artifact mapping, probes, rule catalog, fences) plus the F04 `Bridge` kernel wiring — and **nothing more**; everything cross-cutting (inference, arbitration, verdicts, evidence/taint, render/hash/explain, severity, routing/run-modes) MUST be reused from the kernel, not supplied by the adapter.
- **FR-004**: The adapter MUST be an **observer, not an author**: it MUST expose **no** operation that generates, edits, or rewrites a Spec Kit artifact (`spec.md`/`plan.md`/`tasks.md`/etc.). Governance reads artifacts as data and asserts facts; the anti-goal ("do not replace authored Spec Kit artifacts") is structural, not aspirational.
- **FR-005**: The feature MUST provide a **phase guard** (`whenPhase`) that makes a rule contribute its check only once the supplied `PhaseReached` fact is at or after the named phase, and makes it a definite **not-applicable** (never a failing or unknown verdict) before then — the stateless kernel governing the stateful lifecycle by treating the current phase as a supplied fact.
- **FR-006**: The feature MUST express the workflow's checks as a **catalog of reified rules**, each carrying a `CheckTier` (who decides — `Deterministic`/`AgentReviewed`/`HumanOnly`) and a `Severity` (whether it blocks), and each able to **render to a sentence and explain itself** — replacing the previous design's monolithic, opaque `analyze` pass.
- **FR-007**: The catalog MUST include the **deterministic** checks — `tasks.md` ↔ dependency-topology consistency (every task has deps, deps resolve, acyclic, ids/skills resolve), Constitution-area completeness, surface-baseline/contract currency, and evidence-not-synthetic — and the **judgement** checks as `Opaque`/`AgentReviewed` advisory rules (e.g. "does `plan.md` address every requirement in `spec.md`?", "are the tasks complete and ordered?") carrying a `Question`, and at least one `HumanOnly` check (is the feature in scope / worth doing).
- **FR-008**: The default posture MUST be **advisory in the inner loop**: every inner-loop phase (`constitution`, `specify`/`clarify`, `plan`, `tasks`, `analyze`, `implement`) runs in `Inner`/`Sandbox` run-mode where **no rule blocks** — the full catalog may run as a report across all tiers — and a failing deterministic check informs without stopping the work.
- **FR-009**: The feature MUST provide the **single merge fence** (`mergeFence`) that, at `PhaseReached Merge` in `Gate` run-mode, recomputes the verdict from the base branch and lets the blocking rules bite (constitution complete, evidence not synthetic, contracts current, fenced surfaces verified) — so everything before merge informs and only the boundary enforces.
- **FR-010**: Hard-stopping a specific earlier phase MUST be an **opt-in** addition — a `|> blocking` severity plus a fence on that phase — visible in the constitution and requiring **no kernel change**; the advisory-by-default behaviour is opt-out per rule, not baked in.
- **FR-011**: The feature MUST treat the **constitution as the dial**: which surfaces are fenced and which rules block at merge MUST be configured by the authored constitution (the "enforcement ↔ light" dial), and the Constitution Check MUST be a rule that verifies the dial was filled in honestly (`ConstitutionArea` completeness) — advisory in the inner loop, blocking at merge.
- **FR-012**: The evidence model MUST be the **kernel's F05 model run unchanged**: `TaskState` carries the evidence state, `TaskDependsOn` is the topology, and the `AutoSynthetic` taint MUST propagate down the dependency graph by the kernel's forward-chaining fixed point — `EvidenceGraph` is a derivation over supplied facts, **not a bespoke engine** the adapter ships.
- **FR-013**: The merge fence's **`evidenceNotSynthetic`** rule MUST be a blocking verdict at merge that keeps synthetic-only / synthetic-tainted work from reaching the base branch, and **no disclosure flag or override MUST flip that verdict** — honesty about evidence is enforced separately from the run-mode freedom to iterate.
- **FR-014**: The Spec Kit adapter MUST be **independent and composable**: it references only the F09 SPI and the kernel (never another adapter), it governs this repo standalone, and it MUST **lift unchanged** into a project coproduct when composed with a second domain at one root — the standalone and lifted `(verdict, provenance)` MUST be identical (the F09 faithful-lift guarantee).
- **FR-015**: The SPI-side adapter MUST be **pure** — values and total functions, no I/O, no `Model`/`Msg`/`Effect`, no interpreter (Constitution Principle IV is N/A). **Sensing** the live repository into `SpecKitFact`s (reading `.specify/feature.json`, parsing `tasks.md`/`tasks.deps.yml`, computing artifact-content hashes) is the job of the F08 effects shell and the F12 CLI, **not** this feature; tests feed facts directly.
- **FR-016**: The feature MUST live in a **new component** `FS.GG.Governance.Adapters.SpecKit` separate from the kernel and the SPI, depend on the **SPI** (and through it the kernel) without either depending on it, and keep its dependency footprint **light** (BCL + `FSharp.Core` + SPI + kernel only — no heavy dependency).
- **FR-017**: The public surface introduced by this feature MUST be declared in a **curated signature contract** (`.fsi`) and the API **surface-area baseline MUST be added/updated** to include it (per the repository's surface-drift discipline; Tier 1), with a dependency-hygiene assertion that the adapter references only BCL/`FSharp.Core`/SPI/Kernel.

### Key Entities *(include if feature involves data)*

- **`SpecKitFact` (domain fact union)**: The Spec Kit domain's closed, private vocabulary — `PhaseReached`, `ArtifactPresent`, `TaskState`, `TaskDependsOn`, `SkillBound`, `ConstitutionArea` — the facts the kernel folds over to govern a feature's lifecycle. Owned by this adapter, never copied from or shared with another domain.
- **`Phase`**: The Spec Kit lifecycle stages (`Constitution`, `Specify`, `Clarify`, `Plan`, `Tasks`, `Analyze`, `Implement`, `Merge`) — supplied as a `PhaseReached` fact so the stateless kernel can govern a stateful lifecycle; ordered so `whenPhase` can mean "at or after".
- **`SpecKitArtifact`**: The artifact kinds the workflow produces (`Constitution`, `Spec`, `Plan`, `Research`, `DataModel`, `Contracts`, `Quickstart`, `Tasks`, `TaskDeps`), mapped onto the kernel's structural `ArtifactRef`.
- **Phase guard (`whenPhase`)**: The combinator that makes a rule contribute only at or after a named phase and inert (definite not-applicable) before it — the bridge between the stateless engine and the staged lifecycle.
- **Phase-check rule catalog**: The reified rules — deterministic (graph well-formedness, constitution completeness, contract currency, evidence-not-synthetic), `AgentReviewed` judgement (plan-covers-spec, tasks complete/ordered), and `HumanOnly` (in scope) — each a `CheckRule<SpecKitFact>` carrying a tier, a severity, and a renderable sentence.
- **`mergeFence`**: The single F07 fence at `PhaseReached Merge` that flips run-mode to `Gate`, recomputes from base, and lets the blocking rules (constitution complete, evidence not synthetic, contracts current, fenced surfaces verified) enforce — the one boundary that enforces.
- **Constitution dial**: The constitution-authored configuration of which surfaces are fenced and which rules block at merge (the "enforcement ↔ light" dial), verified for honest completeness by the Constitution Check.
- **Evidence/dependency graph**: The `TaskState` (evidence state) and `TaskDependsOn` (topology) facts over which the kernel's F05 evidence model computes the `AutoSynthetic` taint by forward chaining — a derivation, not a bespoke engine.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: The Spec Kit adapter is **fully specified by exactly the five** SPI components (plus the F04 `Bridge` wiring) — it supplies only those and **reuses 100%** of kernel facilities; the adapter module contains no inference, arbitration, evidence, rendering, hashing, explanation, severity, or routing code, and **no** artifact-authoring operation (FR-003/FR-004).
- **SC-002**: For a `whenPhase P` rule, evaluation over a `FactSet` whose supplied phase is **before** `P` contributes a definite not-applicable for **100%** of such rules (never a failing/unknown verdict), and evaluation at or after `P` contributes the rule's check — proving the stateless kernel governs the stateful lifecycle through the supplied phase fact (FR-005).
- **SC-003**: In **every** inner-loop phase in `Inner`/`Sandbox` mode, **no** rule blocks — a failing deterministic check reports but the route never blocks — and at `PhaseReached Merge` in `Gate` mode the blocking rules flip the route to a blocking gate; the inner-loop-vs-merge distinction is verified across the full catalog (FR-008/FR-009).
- **SC-004**: The `AutoSynthetic` taint propagates down a `TaskDependsOn` chain by the **kernel's** fixed point (no adapter graph code), and `evidenceNotSynthetic` is a **blocking** failure at merge for synthetic-tainted evidence that **no** flag or override flips — verified over supplied evidence/dependency facts (FR-012/FR-013).
- **SC-005**: The Constitution Check is **advisory** in the inner loop and **blocking at merge** for an unfilled/placeholder area, and the merge fence's blocking set is the set the **constitution authored**, not a fixed list in the adapter — verified by varying the dial (FR-011).
- **SC-006**: Each rule in the catalog **renders to a sentence and explains itself** (it produces a human-readable check rendering and an explanation through the kernel's interpreters) — verified for 100% of the catalog, replacing the monolithic `analyze` pass with self-describing rules (FR-006).
- **SC-007**: The Spec Kit adapter **lifts unchanged** into a project coproduct — for **100%** of its rules, the lifted rule's `(verdict, provenance)` over coproduct-wrapped facts is byte-for-byte identical to the standalone original (the F09 faithful-lift guarantee), and the adapter references only the SPI and the kernel, never another adapter (FR-014).
- **SC-008**: The new component adds **no heavy dependency** (references only the SPI, the kernel, `FSharp.Core`, and the BCL), neither the kernel nor the SPI references it, and the API surface-area baseline is recorded — the adapter is exercised entirely through the built component, not private helpers (FR-016/FR-017).

## Assumptions

- This feature corresponds to **F10** (`010-adapter-speckit`) in the dated implementation plan — the **first concrete production adapter** and **domain #1 of Milestone M3 — the adoption bar** ("the kernel is a library, not a platform"). It depends on **F09** (the adapter SPI + `Lift`/`Composition` machinery, already merged) and through it on **F07** (`Fence`/`Route`/run-modes), **F05** (the evidence/taint model), **F04** (`CheckRule<'fact>` = `Check` + `CheckTier` + `Severity` + `Bridge`, plus `toRule` and the cache key), **F03** (`Check`/`Probe`/`ArtifactRef`/`Outcome`), and **F01–F02** (the `Rule`/`FactSet`/`FixedPoint` and verdict algebra) — all already merged. It is composed alongside **F11** (the design-system adapter) and **wired** by **F12** (the CLI's project-level composition root and the sensing of the live repository). It is run by **F08** (the effects shell, already shipped).
- The design is fixed by `docs/governance-design/speckit-in-the-system.md` ("Spec Kit in the system"): the spec-kit adapter, the phase checks as reified rules, run-modes mapped to phases, the constitution as the dial, and the evidence model. The spec-level requirements here are the **behaviours and invariants** that doc describes; the **exact shapes** — the precise `SpecKitFact`/`Phase`/`SpecKitArtifact` cases, the `whenPhase` and `mergeFence` signatures, the specific rule ids, and the `Bridge` wiring — are implementation/design details fixed in the **plan** and the curated **`.fsi`**.
- This is a **pure** feature: values and total folds, no state and no I/O — so **Constitution Principle IV (Elmish/MVU) is N/A**, exactly as for F01–F07 and F09. **Sensing** the live repository into `SpecKitFact`s (reading `.specify/feature.json`, parsing `tasks.md`/`tasks.deps.yml`, hashing artifact content) and **wiring** the adapter into a running loop is the job of the F08 effects shell and the F12 CLI, not this feature; tests feed `SpecKitFact`s directly.
- Governance is the **observer**, Spec Kit is the **harness**: the adapter reads artifacts as data and asserts facts; it **never** generates or rewrites `spec.md`/`plan.md`/`tasks.md`. This anti-goal is enforced structurally — the adapter ships no authoring surface.
- The dependency direction is **adapter → SPI → kernel**, never the reverse; the new component `FS.GG.Governance.Adapters.SpecKit` adds **no heavy dependency** (BCL + `FSharp.Core` + SPI + kernel only). It is exercised through its built public surface (a curated `.fsi`) with a new surface-area baseline and a dependency-hygiene test.
- **Out of scope** (deferred to later features): the concrete **design-system adapter** (F11); the **CLI command surface**, the project-level composition-root *wiring*, and the **sensing** of the live repository (F12 — reading the filesystem, parsing artifacts, hashing content); the **effects edge** (F08, already shipped, which runs the adapter); and any change to the kernel or the SPI (this feature adds adapter code only, the kernel/SPI are unchanged). This feature ships the pure Spec Kit adapter — the five components, the phase guard, the catalog, the merge fence, and the constitution dial — and the example facts the tests feed it.
