# Quickstart: Validating the Golden Enforcement Truth-Table Fixtures

This guide shows how to generate, bless, and validate the committed fixtures. It is a run/validation
guide — the row-by-row contract lives in [contracts/truth-table-format.md](./contracts/truth-table-format.md)
and [contracts/audit-snapshot-set.md](./contracts/audit-snapshot-set.md); the entities are in
[data-model.md](./data-model.md).

## Prerequisites

- .NET `net10.0` SDK (repo standard).
- The merged cores build (they already do): F014/F015/F017/F023/F024/F025. This row adds **no** `src/`
  code, so a clean `dotnet build` of the solution is unchanged except for the new test project.

## Generate / re-bless the committed fixtures

The fixtures are generated by the drift-guard test in re-bless mode (mirrors the repo's `BLESS_SURFACE=1`
surface-baseline idiom). Run this once to write the initial committed files, and any time an **intended**
core change should update them:

```bash
BLESS_FIXTURES=1 dotnet test tests/FS.GG.Governance.EnforcementFixtures.Tests
```

This writes:

- `fixtures/enforcement/truth-table.md` — the 240-row primary cross-product + the route-class section.
- `fixtures/enforcement/audit-snapshots/*.audit.json` — one document per blocking-altering scenario.

Review the diff before committing: an intended enforcement change shows up here as the precise set of
changed rows/snapshots (that diff *is* the review artifact, SC-005).

## Validate (the drift guard — the normal CI path)

```bash
dotnet test tests/FS.GG.Governance.EnforcementFixtures.Tests
```

Expected results when the committed fixtures match the live cores:

- **Completeness (SC-001)** — the primary table has exactly 240 data rows and 240 distinct
  `(base,maturity,mode,profile)` keys (= 2×5×6×4); no missing, no duplicate.
- **Drift guard (SC-002/SC-003)** — the regenerated `truth-table.md` bytes equal the committed bytes;
  regenerating twice yields identical output (determinism).
- **Route-class (FR-003)** — the routine row is `out-of-scope` + `(none)` and never default-denies (also
  under `release`/`release`); the fenced row is `routed:<domain>` + `(none)`; the unknown-governed-path row
  carries an explicit finding id.
- **Snapshots (FR-007–FR-010, SC-004)** — each committed `*.audit.json` equals
  `AuditJson.ofShipDecision (Ship.rollup …)` for its scenario; every blocking-altering dial (maturity, base
  severity, profile, run mode) is represented; each relaxed-blocker snapshot shows both base and effective
  severity + reason (no-hide).

## Confirm the drift guard actually guards (manual smoke)

To prove the guard fails on an unintended change (do **not** commit this):

1. Hand-edit one reason cell in `fixtures/enforcement/truth-table.md`, or flip an `effective` token.
2. Run `dotnet test tests/FS.GG.Governance.EnforcementFixtures.Tests` — the drift-guard test fails with a
   readable diff naming the file and the `BLESS_FIXTURES=1` re-bless command (SC-003).
3. `git checkout -- fixtures/enforcement/` to restore; re-run — it passes again.

## Confirm nothing else changed (Tier-2 guarantee)

```bash
dotnet build FS.GG.Governance.sln
dotnet test                      # whole solution
git status --porcelain surface/ src/   # expect: no changes
```

the only additions are `tests/FS.GG.Governance.EnforcementFixtures.Tests/`, the `fixtures/enforcement/`
goldens, the solution entry, and a README pointer.
