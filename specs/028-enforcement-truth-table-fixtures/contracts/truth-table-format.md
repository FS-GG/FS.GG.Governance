# Contract: Golden Truth-Table File Format (`fixtures/enforcement/truth-table.md`)

The committed truth table is a byte-stable Markdown document. This contract fixes the byte-level rules the
drift guard enforces (FR-005, FR-006); any change to these rules is a deliberate re-bless. The **values** in
the table come verbatim from the merged cores (FR-002) — this contract governs only their *rendering*.

## File-level rules

- Encoding: **UTF-8, no BOM**. Newlines: **`\n`** (LF) only. **No trailing whitespace** on any line. The
  file ends with exactly one trailing `\n`.
- The file has two sections in this fixed order: (1) the **primary cross-product table**, (2) the
  **route-class table**, each under a `##` heading. A short `#` title and a one-line "generated — do not
  edit by hand; regenerate with `BLESS_FIXTURES=1 dotnet test`" note precede them.
- Determinism: the file content is a pure function of the merged cores. No clock, host path, environment,
  username, or enumeration-order influence (FR-004).

## Dial-token spelling (fixed)

Tokens are the lowercase kebab spelling of each closed value, fixed here so a rename in rendering is caught:

| Type | Value → token |
|---|---|
| `Severity` | `Advisory`→`advisory`, `Blocking`→`blocking` |
| `Maturity` | `Observe`→`observe`, `Warn`→`warn`, `BlockOnPr`→`block-on-pr`, `BlockOnShip`→`block-on-ship`, `BlockOnRelease`→`block-on-release` |
| `RunMode` | `Sandbox`→`sandbox`, `Inner`→`inner`, `Focused`→`focused`, `Verify`→`verify`, `Gate`→`gate`, `Release`→`release` |
| `Profile` | `Light`→`light`, `Standard`→`standard`, `Strict`→`strict`, `Release`→`release` |
| `RoutingResult` | `Routed(d,…)`→`routed:<domain>`, `UnmatchedInRoot`→`unmatched-in-root`, `OutOfScope`→`out-of-scope` |
| `FindingId` | rendered via `Findings.findingIdToken`; `(none)` when no finding |

> The run-mode and profile token maps deliberately match the F023 `recognizeMode`/`recognizeProfile`
> spellings so the table reads in the same vocabulary callers type. Where a token spelling must be derived
> from the core (e.g. `findingIdToken`), the core's function is used — not a re-spelling.

## Primary cross-product table

- Header row, then a `|---|…|` separator row, then exactly **240** data rows.
- Columns, in this fixed left-to-right order:

  `| base | maturity | mode | profile | effective | reason |`

- **Row order** is the fixed nested iteration, outermost → innermost:
  `base` (advisory, blocking) → `maturity` (observe, warn, block-on-pr, block-on-ship, block-on-release) →
  `mode` (sandbox, inner, focused, verify, gate, release) → `profile` (light, standard, strict, release).
  (Innermost dial varies fastest.) This order is part of the pinned bytes.
- `effective` = `(deriveEffectiveSeverity input).EffectiveSeverity` token; `reason` =
  `(deriveEffectiveSeverity input).Reason` rendered verbatim. The reason is treated as pinned outcome text
  (Edge: reason-text stability) — a reword trips the guard.
- Reason text that contains a `|` MUST be escaped as `\|` so the Markdown table stays well-formed; this is
  the only transformation applied to reason text, and it is byte-stable.

## Route-class table

- Header, separator, then one row per route-class scenario (D6): at minimum `routine`, `fenced`,
  `unknown-governed-path`, plus the protected-surface unknown variant.
- Columns, in this fixed order:

  `| class | example path | route outcome | finding | note |`

- Values come from `Routing.route` and `Findings.findUnknownGovernedPaths` over real `TypedFacts`; the
  `note` column states the demonstrated property in fixed wording (see data-model). The `routine` row's
  note records that it selects nothing and never default-denies even under the strictest dials.

## Guard behavior

- The drift guard regenerates the whole file and asserts byte-equality with the committed copy; on
  mismatch it fails with a readable per-line diff naming `fixtures/enforcement/truth-table.md` and the
  `BLESS_FIXTURES=1` re-bless command (FR-006, SC-003).
- A separate completeness assertion counts the primary data rows (must equal 240 = product of dial
  cardinalities) and the distinct four-tuple keys (must also equal 240 — no duplicates), so a missing or
  doubled combination fails independently of the byte diff (SC-001).
- Adding a future dial value changes the generated bytes and the count, failing the guard until the
  committed copy is re-blessed (FR-012, Edge: cross-product growth) — the table can never silently
  under-cover a dial.
